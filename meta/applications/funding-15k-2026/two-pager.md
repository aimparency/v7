# Aimparency

## Git-native intent and dependency infrastructure for AI-assisted development

AI coding agents can produce changes quickly, but repositories mainly preserve
what changed. They do not reliably preserve why the work mattered, which goals
it served, what depended on it, or whether the result fulfilled the original
intent.

Aimparency adds that missing layer. It stores aims, dependencies, priorities,
status, evidence, and agent-readable context next to the repository. Humans
remain able to inspect and revise the plan while agents discover, execute, and
report work through the same graph.

## Working Product

The current local-first prototype includes:

- a graph and structured list interface for aims and dependencies;
- Git-native project state under `.bowman/`;
- MCP access for coding assistants;
- prioritized aim discovery using value and cost flow;
- local Claude, Gemini, Codex, and loop-worker integration;
- semantic aim search and local semantic code indexing;
- resumable human and external dependencies;
- structured experiments, evidence, and belief updates.

This is running software, not a presentation prototype.

## Initial Users

The initial users are technical founders, open-source maintainers, and
AI-native engineering teams coordinating several agents or long-running
projects. The immediate question is not total market size. It is whether these
teams experience enough coordination loss to pay for a clearer intent layer.

Aimparency will test that through paid design partnerships rather than assume
the answer.

## Differentiation

Task boards organize assignments. Chat histories preserve conversations. Git
preserves code history. Aimparency connects durable intent to implementation:

- local-first and repository-owned;
- versionable and inspectable as ordinary project data;
- graph-based rather than a flat ticket queue;
- model-independent;
- designed for cooperation and authorization boundaries, not unattended action
  at any cost.

## Six-Month Plan

1. Interview at least 20 qualified AI-native teams.
2. Sign three paid design partnerships or establish clear rejection evidence.
3. Make intent-to-code provenance reliable across real repositories.
4. Turn the autonomous loop into a measurable experiment and delivery system.
5. Publish the open-source core and document a sustainable hosted/support path.
6. Use observed retention and willingness-to-pay evidence to choose grants,
   revenue financing, or a larger angel round.
